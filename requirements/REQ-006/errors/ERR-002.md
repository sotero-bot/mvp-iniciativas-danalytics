## Descripción

Al desactivar (borrar) un paso desde la vista de administración de actividades, el contador de pasos en `/runner/:token` seguía mostrando el número anterior. Por ejemplo, desactivar un paso con 9 activos mostraba "9 pasos" en lugar de "8 pasos".

## Causa raíz

El endpoint `GET /execution/:token` consultaba la tabla `PasoActividad` **sin filtrar por `activo: true`**, devolviendo todos los pasos incluyendo los desactivados. El frontend calcula el contador directamente como `data.pasos.length`, por lo que el valor no se actualizaba.

El endpoint de administración (`admin-actividades.controller.ts`) sí filtraba correctamente con `where: { activo: true }`, lo que creaba una inconsistencia entre ambas vistas.

## Solución aplicada

Se añadió `where: { activo: true }` a la query de `pasos` dentro del `prisma.actividad.findUnique` del execution controller:

```typescript
// apps/api/src/modules/execution/interfaces/execution.controller.ts
pasos: {
  where: { activo: true },  // ← añadido
  orderBy: { orden: 'asc' },
  include: { preguntas: { where: { activo: true }, orderBy: { orden: 'asc' } } },
},
```

## Notas

- El filtro de `preguntas` por `activo: true` ya existía; faltaba el equivalente para `pasos`.
- El fix también evita que pasos desactivados aparezcan en la navegación del runner (StepPills, "Paso N de M").
