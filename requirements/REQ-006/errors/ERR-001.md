## Descripción

Al acceder a `/runner/enlace/:token` en modo desarrollo, se creaban **dos instancias** (`InstanciaActividad`) en lugar de una.

## Causa raíz

React 18 **StrictMode** (activo solo en desarrollo) ejecuta cada `useEffect` dos veces: mount → cleanup → remount. El `useEffect` que llama a `POST /execution/enlace/:token/sesion` no tenía protección contra esta doble ejecución, por lo que el endpoint se invocaba dos veces y se creaban dos registros en BD.

## Solución aplicada

Se añadió un `useRef` guard (`hasStarted`) al inicio del efecto. React 18 StrictMode **preserva** el valor del ref entre el mount/unmount/remount simulado, por lo que la segunda ejecución del efecto encuentra `hasStarted.current = true` y retorna sin hacer el fetch.

```tsx
const hasStarted = useRef(false);

useEffect(() => {
    if (hasStarted.current) return;
    hasStarted.current = true;
    // ... POST request
}, [token, navigate]);
```

## Notas

- **No usar** `AbortController` combinado con el ref guard: se contrarrestan (el abort cancela la primera llamada y el ref bloquea la segunda, resultando en que ninguna llega al servidor y la página queda atascada en "Preparando tu sesión...").
- Este bug **no ocurre en producción** ya que StrictMode no está activo en el build de producción.
